## 2024-03-08 - Fast CSV Generation with Built-in Library

**Learning:** When exporting large volumes of logs/data via Python's `csv` module (e.g. JSONL to CSV), using `csv.DictWriter` adds per-row overhead to reconstruct/map dictionary values back into lists aligned with the headers. Additionally, aggressively calling methods like `strip()` and `json.loads()` on every line, when we can short-circuit non-JSON payloads by just checking `line[0] == '{'`, dramatically improves iteration speed. Finally, inlining validation checks (e.g., using `.isspace()` or checking characters in a string like `value[0] in "='+-@"`) is much faster than running full `.startswith()` and `.lstrip()` commands over strings that don't need sanitization.
**Action:** Always favor `csv.writer` with pre-computed key/list mappings when writing large amounts of data, except when flexibility constraints heavily dictate otherwise. Use index character lookups on strings as a fast-path for expensive validation logic to avoid function call overhead.

## 2024-11-20 - [Avoid Micro-Optimizing High-Cardinality String Caching]
**Learning:** Adding `@functools.lru_cache` to simple string sanitization functions (like `_sanitize_formula` in `log_export.py`) is a premature optimization that can actually degrade performance. In a log export context, string values (timestamps, correlation IDs, unique log messages) have extremely high cardinality. The `lru_cache` suffers a high miss rate, and the overhead of hashing, dictionary lookups, and maintaining the LRU linked list makes the script slower and uses more memory than just executing the simple string operations. Also `type(x) is str` instead of `isinstance(x, str)` is an anti-pattern.
**Action:** When optimizing tight loops, prefer hoisting global function and object attribute lookups to local variables (which use faster `LOAD_FAST` opcodes). Avoid `lru_cache` unless you can mathematically guarantee a very high cache hit rate. Stick to `isinstance()` for type checking.
