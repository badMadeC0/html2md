## 2024-05-08 - Added Request Size Limits to Prevent Denial of Service
**Vulnerability:** The CLI fetched remote URLs directly into memory without any size constraints. A malicious or misconfigured server returning a multi-gigabyte HTML response would cause an Out of Memory (OOM) error, creating a Denial of Service (DoS) vulnerability.
**Learning:** Even simple CLI fetch tools are susceptible to resource exhaustion attacks if response sizes are unbounded, especially since `requests.get` without `stream=True` buffers the entire response in memory.
**Prevention:** Always use `stream=True` when downloading untrusted resources, and enforce a strict upper limit on downloaded bytes.

## $(date +%Y-%m-%d) - Prevent SSRF in `html2md` URL Fetching
**Vulnerability:** The CLI fetched arbitrary user-supplied URLs via `requests.Session.get()` without resolving and validating the hostname IP address. This allowed Server-Side Request Forgery (SSRF) where an attacker could provide URLs like `http://127.0.0.1` or `http://localhost` to access internal services and bypass firewall rules.
**Learning:** Checking the string URL is insufficient for SSRF protection because attackers can provide domain names that resolve to internal IPs. Using `socket.gethostbyname()` is also insufficient because it only handles IPv4 and might ignore IPv6 addresses (like `::1`). `socket.getaddrinfo()` must be used to reliably resolve both IPv4 and IPv6 addresses.
**Prevention:** Always validate that hostnames do not resolve to private, loopback, or link-local IP addresses using `socket.getaddrinfo()` and `ipaddress.ip_address` before passing them to an HTTP client. Note that for robust systems, IP validation should ideally happen *after* DNS resolution at the HTTP connection level to prevent DNS Rebinding (TOCTOU) attacks, but checking beforehand provides a strong first layer of defense.
